SharePoint Site & List Setup

Setup
1.1
TOOL · SharePoint Admin (sharepoint.com)
Create the Intake SharePoint Site
Create a dedicated private Team Site to house all intake list data — keeps permissions clean and separate.
▾
  • Go to sharepoint.com → + Create site → Team site
  • Name: IntakeRequests-[OrgName] · Privacy: Private
  • Add Site Owners (admins) and Site Members (all staff who can submit requests)
  • Inside the site → + New → List → Blank list → name: IntakeRequests
  • Enable versioning: List Settings → Versioning Settings → Create a version each time you edit an item
  • Set item-level permissions: List Settings → Advanced Settings → Read items that were created by the user
⚠️

Item-level permissions limit what submitters can see. The Power Automate service account and approvers need Contribute access or higher to update Status and ApprovedBy columns after approval.

💡

One site can hold multiple intake lists (IT Requests, HR Requests, Facilities). Use one Canvas App with a RequestType dropdown to route to different lists, or one list with a category column.

1.2
TOOL · SharePoint List · Column Settings
Create List Views for Approvers & Submitters
Configure filtered views so each role sees only what they need — no extra apps or dashboards required.
▾
  • My Requests view — filter: RequestedBy = [Me] · Default view for submitters
  • Pending Approval view — filter: Status = Submitted · For approvers
  • All Requests view — no filter · Admin-only access via SharePoint permissions
  • By Department view — group by Department column · For admin reporting
💡

Power Apps will bypass these views and filter records directly via formula. The views are for admins and approvers who review the list in SharePoint directly.

02

Create the Canvas App

Build
2.1
TOOL · Power Apps Studio (make.powerapps.com)
Create Canvas App & Connect SharePoint
Build the intake form as a Canvas App backed by SharePoint — all standard connector, zero premium licensing.
▾
  • Go to make.powerapps.com → + Create → Canvas App from blank
  • Name: IntakeRequest-App · Format: Tablet (better for multi-field forms)
  • Data panel → Add data → SharePoint → enter your site URL → select IntakeRequests list
  • Confirm the data source appears in the left Data panel before building screens
  • Plan 3 screens: Screen 1 = Request Details · Screen 2 = Review & Submit · Screen 3 = Confirmation
✅

SharePoint is a standard connector. Every M365 Business Premium user can run this app at no extra cost — no Power Apps per-user plan required.

2.2
TOOL · Power Apps Studio · Formula Bar
Submit Button — Patch() to SharePoint List
The submit button writes a new record to the SharePoint list using Power Apps' Patch() function.
▾

Set the OnSelect property of your Submit button:

// OnSelect of Submit button — writes to SharePoint list Patch( IntakeRequests, // SharePoint list data source name Defaults(IntakeRequests), // Creates a new item { Title: txtTitle.Text, RequestType: { Value: ddRequestType.Selected.Value }, Description: txtDescription.Text, Priority: { Value: ddPriority.Selected.Value }, Department: { Value: ddDepartment.Selected.Value }, TargetDate: dtpTargetDate.SelectedDate, EstimatedCost: Value(txtCost.Text), Status: { Value: "Submitted" }, RequestedBy: { Claims: "i:0#.f|membership|" & User().Email, DisplayName: User().FullName, Email: User().Email } } ); Navigate(scrConfirmation, ScreenTransition.Fade);
⚠️

Person/Group columns require the Claims string format with i:0#.f|membership| prefix. Copy exactly — a wrong format causes a silent write failure.

Build App Screens & Formulas

Build
4.1
TOOL · Power Apps Studio
Home Screen — "My Requests" Gallery
Show the logged-in user's submitted requests with live status badges — filtered directly from SharePoint.
▾

Set the Items property of a vertical gallery:

// Gallery Items — show current user's requests, newest first SortByColumns( Filter( IntakeRequests, RequestedBy.Email = User().Email ), "Modified", SortOrder.Descending )

Dynamic status badge colour — set the Fill of a rectangle inside the gallery:

Switch( ThisItem.Status.Value, "Submitted", ColorValue("#1a5cb0"), "Approved", ColorValue("#2a7a4f"), "Rejected", ColorValue("#b02a2a"), "In Review", ColorValue("#b06a00"), ColorValue("#9ca3af") // Draft )
4.2
TOOL · Power Apps Studio
Approver View Screen — Pending Queue
A separate screen for approvers to see all pending submissions across the organisation.
▾
// Approver gallery — show all Submitted items SortByColumns( Filter( IntakeRequests, Status.Value = "Submitted" ), "Created", SortOrder.Ascending // Oldest first = FIFO queue ) // Show approver screen only if user is in the approver group // On App OnStart, set: varIsApprover = !IsBlank(LookUp(ApproverGroup, Email = User().Email)) // ApproverGroup = a separate SP list with one column: Email
💡

Maintain a small SharePoint list called ApproverGroup with one Email column. Use LookUp(ApproverGroup, Email = User().Email) in App OnStart to set a varIsApprover variable — show or hide the approver nav item accordingly.

05

Power Automate Approval Flow

Automate
5.1
TOOL · Power Automate (make.powerautomate.com)
Build the SharePoint-Triggered Approval Flow
Standard-connectors-only flow: SharePoint trigger → Approvals → Teams notification → SharePoint update.
▾
  • Go to make.powerautomate.com → + Create → Automated cloud flow
  • Trigger: "When an item is created" → SharePoint → Site: your intake site → List: IntakeRequests
  • Add Condition: Status Value is equal to Submitted (prevents triggering on Draft saves)
  • Yes branch → Start and wait for an approval → Type: Approve/Reject - First to respond
  • Assigned To: your approver's email (or dynamic group email based on Department column value)
  • Title: Intake Request: [Title] · Details: compose request type, submitter, description
  • After approval step: add Condition on Outcome — branch Approved vs Rejected
✅

This flow uses only SharePoint, Approvals, and Microsoft Teams connectors — all standard. Fully covered by M365 Business Premium. No premium connector licence required.

5.2
TOOL · Power Automate · Branches
Update Status & Notify Submitter on Decision
After approval decision, write the outcome back to SharePoint and send a Teams notification to the submitter.
▾
BranchSP UpdateNotification
ApprovedStatus = Approved, ApprovedBy = approver email, ApproverComments = commentsTeams message to submitter: "Your request [Title] has been approved."
RejectedStatus = Rejected, ApproverComments = rejection reasonTeams message to submitter with rejection reason from approver
💡

Add a parallel Send an email (V2) action (Office 365 Outlook — standard connector) alongside the Teams notification. Approvers and submitters receive both a Teams card and an email confirmation.

Teams Deployment & Publishing

Deploy
7.1
TOOL · Power Apps Studio + Teams Admin Center
Publish App & Pin as Teams Tab
Surface the intake app directly in Teams — users never need to leave Teams to submit a request.
▾
  • In Power Apps Studio → Save → Publish
  • On the app detail page → Add to Teams → download the Teams app package (.zip)
  • In Teams Admin Center → Teams apps → Manage apps → Upload new app → upload the .zip
  • Create a Setup Policy → pin the intake app for all staff or targeted groups
  • Alternatively: in Teams channel → + Add a tab → Power Apps → select the intake app
  • Test SSO: open the tab — users should sign in automatically, no extra prompts
  • Run an end-to-end test: submit a request, confirm it appears in SharePoint list, confirm approval flow triggers
✅

Power Apps in Teams uses the same Entra ID session — SSO is seamless. No extra authentication steps for users already in Teams.

08

Scale — New Request Type Onboarding

Scale
8.1
TOOL · SharePoint + Power Apps — config only
5-Step New Request Type Checklist
No new infrastructure needed. Adding a new intake category is just configuration — under 30 minutes per type.
▾
  • 1 — Register: Add the new category to the RequestType Choice column in the SharePoint list
  • 2 — Routing: Update Power Automate flow — add condition branch for new category to route to the correct approver email or group
  • 3 — App update: If category needs unique form fields, add a conditional section in the Canvas App using If(ddRequestType.Selected.Value = "NewType", ...)
  • 4 — Permissions: Confirm the new approver has Contribute access to the SharePoint list
  • 5 — Test: Submit a test request of the new type · verify approval routes correctly · confirm status updates in list
🚀

Target: under 30 minutes per new intake type. All infrastructure — SharePoint site, Canvas App, approval flow — is already live. You're only registering a new dropdown choice and routing logic. No new deployments.

⚠️

If you are approaching 300 users on Business Premium, plan a migration to M365 E3 or E5 before hitting the cap. The SharePoint architecture scales directly to E3/E5 without any rework.

Selected Architecture: Power Apps Canvas App → SharePoint List

For internal intake requests (IT, HR, Facilities, General) on M365 Business Premium where no additional licensing spend is acceptable, Option 3 is the correct architecture. For non-financial intake with standard row isolation, SharePoint is sufficient.
